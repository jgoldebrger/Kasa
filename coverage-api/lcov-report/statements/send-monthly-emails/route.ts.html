/**
 * POST /api/statements/send-monthly-emails
 *
 * Cron-or-admin entry point that emails the previous month's statement
 * to every family with an email on file.
 *
 * Previously processed every family inline in a single request. On a
 * large org (> ~50 families) this overran the 10s Vercel serverless
 * budget and the job died mid-way — leaving some families emailed and
 * the rest not. The Statement rows for the skipped families would also
 * be missing because each family's send call lazily-creates a
 * Statement, so a partial run created an audit gap.
 *
 * Now enqueues an `EmailJob` (kind: 'statements') and fires the existing
 * chunked worker at `/api/statements/send-emails/worker`. The worker:
 *   - claims a batch of family ids atomically (no double-send under
 *     concurrent ticks)
 *   - delegates each family's actual send to `sendOneFamilyStatement`
 *     (same helper the interactive bulk-send uses, so cycleCharges and
 *     org currency/locale are threaded through correctly)
 *   - self-triggers the next batch until pending is empty
 *
 * Response is 202 with `{ jobId, totalFamilies, status: 'queued' }`.
 * The cron caller doesn't poll — it's fire-and-forget by design.
 * Interactive admin callers can poll `/api/statements/send-emails/status?jobId=…`.
 */
